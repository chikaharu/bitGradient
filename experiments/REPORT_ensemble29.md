# LLM-less 勾配降下法による Rust コンパイルエラー自動修正
## ― コンパイラをオラクルとした離散探索の記録 ―

**ensemble29-v3 / experiment-55**  
2026-04-15 現在  ·  最高スコア **51 / 56**  ·  正答率 **91.1 %**

---

## はじめに ― この実験が問うもの

大規模言語モデル (LLM) を使わずに、Rust のコンパイルエラーを自動修正できるか。

問いはシンプルだが、含意は深い。LLM はパターンを「記憶」することで正解に辿り着く。それに対してこの実験が試みるのは、**記憶を持たない探索機械**が、コンパイラという「Yes/No オラクル」の応答だけを手がかりに、記号空間を下り坂で歩けるか、という問いである。

損失関数は存在しない。微分可能な関数も存在しない。あるのは「エラーが減ったか、増えたか」という二値の信号だけだ。それを連続最適化の類比として捉え直し、**nibble (u4) ビット演算**を勾配の代わりに用いる——それがこのプロジェクトの核心である。

## アーキテクチャ概観

```
入力: Rust ソースファイル (コンパイルエラー付き)
         │
         ▼
   rustc (コンパイラオラクル) ─── エラースパン / extra トークン抽出
         │
         ▼
  nibble フィンガープリント ──── 各トークンを 1024-bit ベクトルに射影
         │
         ▼
  5種変異候補生成 ─────────── SNV / INSERT / DEL_F / DEL_B / CNV
         │
         ▼
  多目的スコアリング ──────── pen - α·sim - β·rcf2 - γ·cs - δ·head
         │
         ▼
  コンパイラオラクル評価 ───── 上位 40 候補を実際にコンパイル
         │
         ▼
  最良候補採用 → 次ステップへ (最大 25 ステップ)
```

## スコアリング関数の進化

### ensemble29-v3 最終形

```
score(mut) = penScore
           − α · nibbleSim       (α = 1.0)
           − β · rcf2Score       (β = 0.5)
           − γ · csScore         (γ = 0.3)
           − δ · headScore       (δ = 0.2)
```

| 項 | 意味 | 値域 |
|----|------|------|
| `penScore` | ペナルティマスク上での採点 (過去失敗ビットを減点) | [0, ∞) |
| `nibbleSim` | nibble XNOR 類似度 (正方向=良い候補) | [0, 1] |
| `rcf2Score` | RC⊗C テンソル (行列二乗ノルム型の文脈一致度) | ℝ |
| `csScore` | CS 直交行列スコア (位置埋め込み一致度) | ℝ |
| `headScore` | 先頭エラートークン 4×4 u4 行列スコア | [-8, 8] |

## 5 種変異操作

```
SNV    (Single Nucleotide Variant)
  → rem → add  (トークン 1 個置換)
  → edit-dist = 1 のとき SNV_BONUS = -1.0 (優先)

INSERT
  → エラースパン行の先頭に候補トークンを行挿入

DEL_F  (Forward Deletion)
  → extra[0] (スパン先頭エラートークン) の最初の出現を削除

DEL_B  (Backward Deletion)
  → extra[-1] (スパン末尾エラートークン) の最後の出現を削除

CNV    (Copy Number Variant)
  → rem → rem rem  (inline 複製)
```

いずれも DNA 変異の命名体系を借用している。コードを「配列」として扱い、一文字の変異 (SNV)・挿入 (INSERT)・欠失 (DEL) の三操作が中心となる点は、生物学的変異の分類と完全に対応している。

## headScore: 4×4 u4 行列による自己埋め込み

スパン先頭エラートークン `headTok = extra[0]` の nibHex 先頭バイト上位4bit `(b₀, b₁, b₂, b₃)` から 4×4 双極行列 **H** を構成する:

```
row[0] = [b₀, b₁, b₂, b₃]   ← nibble 原値
row[1] = [b₁, b₂, b₃, b₀]   ← 左ローテーション 1
row[2] = [b₂, b₃, b₀, b₁]   ← 左ローテーション 2
row[3] = [b₃, b₀, b₁, b₂]   ← 左ローテーション 3
各要素: 0 → −1, 1 → +1  (双極化)
```

XOR 自己埋め込みで行列を調整:

```
xorHash[j]    = row[0][j] ⊕ row[1][j] ⊕ row[2][j] ⊕ row[3][j]
H_embed[i][j] = H[i][j] × sign(xorHash[j])
headScore     = Σᵢ ( ‖H_embed[i]‖ · ⟨H_embed[i], addVec⟩ / 4 )
              ∈ [−8, 8]
```

`addVec` は候補トークン `addBits[0]` の下位4bitから生成した双極4次元ベクトルである。

**直感的な意味:** エラートークンのビットパターンと候補トークンのビットパターンが「回転的に整合している」ほど headScore が高くなり、候補のスコアが下がる (= より優先されやすい)。

| typeid-consistency.rs | 3 | 2 | **0.673** | max | Step 6 まで pen 漸増、9.1% で飽和。H=2 から 1 減らせない。|
| type-ascription.rs | 2 | 2 | **1.000** | max | H=2 (flat)。初期から探索空間がゼロ。|
| type-unsatisfiable.rs | 2 | 2 | **1.000** | stop@12 | 変異で H 変化なし → 12ステップで早期終了。|
| verbose.rs | 2 | 2 | **1.000** | stop@10 | 変異で H 変化なし → 10ステップで早期終了。|

**combined 平均 AUC = 0.788**

### ペナルティ飽和 ― 天井の正体

`type-check-defaults.rs` と `typeid-consistency.rs` の共通パターン:

```
Step 0  : pen =  0.0%    H = 8 (or 3)
Step 1  : pen =  2.6%    H が 2 へ急落 ← 最大の改善
Step 5  : pen =  8.7%    以降変化なし
Step 10 : pen =  9.2%    ← 飽和点
Step 10-25: pen ≈ 9.2%   H = 2 から脱出できない
```

1024-bit のペナルティマスクに約 94 bit (9.2%) が立った状態で、**有効な候補空間が消耗しきる**。これ以上の探索は同じビットパターンを繰り返すだけであり、ランキングが収束して局所極小に陥る。

### type-ascription / type-unsatisfiable / verbose ― 別種の困難

これら3件は H₀=2 かつ AUC=1.0 (完全な flat)。つまり初期状態から一切改善されなかった。

- **エラーが意味論的** (型の不整合、satisfiability 問題) であり、トークン置換・挿入・削除では構造的に解決不能。
- `type-ascription` はそもそも Rust の実験的構文 (`x: Type` 式中型注釈) に起因しており、構文レベルの書き換えが必要。

## 技術的洞察

### なぜ LLM なしで 51/56 を達成できるのか

1. **コンパイラが意味論的フィードバックを与える**: `rustc` のエラーメッセージは「どこが」「なぜ」間違いかを教えてくれる。この信号がオラクルとして極めて高品質である。

2. **nibble XNOR が局所構造を捉える**: u4 (4-bit) 単位のビット演算は、トークンの文字レベル類似性を高速に近似する。完全な意味理解は不要であり、「似たトークン」を素早く列挙するための十分な近似として機能する。

3. **ペナルティマスクが記憶を持つ**: 過去の失敗を 1024-bit ベクトルとして蓄積し、同じ過ちを繰り返さない。LLM のコンテキスト窓の代わりに、ビット列が「短期記憶」を担う。

4. **RC⊗C + CS が構造を読む**: トークン列のテンソル積と位置埋め込みにより、単純な類似度を超えた「文脈整合性」を評価できる。

### なぜ 51/56 で天井を打つのか

1. **ペナルティ飽和**: 有限の 1024-bit マスクが探索空間の消耗を加速する。

2. **意味論的エラーへの無力さ**: ビット演算はシンタックスの近傍しか探索できない。型システムの制約を突破する変異は生成できない。

3. **非連続な損失地形**: H=2→H=1 の遷移は、複数トークンの同時変換を必要とする可能性がある。greedy な 1 ステップ探索では谷が見えない。

## 次なる壁をどう越えるか

### 方向 A: ペナルティマスクのリセット戦略
- 飽和率 > 8% になった時点で、ペナルティの「半減期」を設ける
- 古い失敗の重みを指数減衰させ、探索空間を再生する

### 方向 B: 多トークン同時変異
- H=2→H=1 の壁は「2トークンの同時置換」が必要な可能性
- 2-SNV の組み合わせ爆発を抑えるため、候補数を絞って探索

### 方向 C: 意味論的変異
- `type-ascription` 系はトークン列ではなく AST 変換が必要
- `syn` crate で AST を解析し、型注釈の位置を意味的に操作する

### 方向 D: 確率的サンプリング
- ペナルティマスクをソフトな重み分布に変換し、Boltzmann サンプリング
- 温度パラメータ T を徐々に下げながら探索 (模擬焼きなまし)

## おわりに

91.1% ——これは LLM なしで、ビット演算と `rustc` の応答だけを使って到達した数字だ。

この実験が示したのは、**コンパイラは優れた教師である**という事実である。文法エラーの大半は「どのトークンを、どのトークンに変えれば良いか」という局所的な問題であり、その局所性を nibble ビット空間が近似できる。LLM が不得意とする「なぜそのトークンが正しいのか」の理解は不要だった。

残り5件は「局所性の限界」を示すエラーだ。型システムの制約、実験的構文、意味論的不整合——これらは文字列の置換では解けない。しかしその壁の形が、この実験を通じてはっきりと見えてきた。

**形が見えたということは、越えられるということだ。**
